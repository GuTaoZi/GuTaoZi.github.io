Task 1 Database Design

Requirements

  • The tables created should satisfy the three normal forms;

  • Use the primary key and the foreign keys to indicate important attributes and relationships about your data. Every row in each table should be uniquely identified by its primary key;

  • Every table should be involved in a link. No isolate tables included;

  • Your design should contain no circular links;

  • Each table should always have at least one mandatory (“Not Null”) column (including the primary key but not the system-generated ID column);

  • Tables with no other unique columns than possibly a system-generated ID is not allowed;

  • Use appropriate types for different fields of data;

  • Your design should better be as easy to expand as possible.

Many-to-Many Relations

通过观察数据发现，同一门course会因instructor不同而开设不同section，同一section也会因上课时间不同而分为不同class，因此课程采用了course-section-class三张表进行存储以避免冗余，满足三大范式。

student和section之间、instructor和section之间也存在多对多关系，因此建立了selection和teach表进行存储以满足三大范式。

Prerequisite Relation Storage

在本项目数据库的设计中，先修课关系由两张表存储：prerequisite_group和prerequisite_combination。一门非叶子结点课的先修课之间有两种关系：AndPrerequisite和OrPrerequisite，观察数据得知先修课关系树的层数并不高，因此可以通过列举一门课的所有先修课组合的方式来存储，这种存储方法便于操作者理解和维护，能规避建树存储时在本地所需的递归等复杂操作，用空间换时间，分析得知对于course_info.json中<1000门课程而言，空间是绰绰有余的。

在prerequisite_group表中，存储了一门课对应的先修组和先修组内课程总数的计数；在prerequisite_combination表中，则存储了各先修组内包含的课程，统计学生是否满足先修时，先通过prerequisite_group确认所有可能的group_id，通过join操作和prerequisite_combination建立联系，当学生已经通过的课程计数满足某一组要求时，判定为满足先修要求，逻辑非常直观，嵌套层数较低，且避免了大量本地运算，节约了带宽。
